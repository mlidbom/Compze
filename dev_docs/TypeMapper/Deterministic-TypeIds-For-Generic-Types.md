# Deterministic TypeIds for Generic Types

> **No backward compatibility constraints.** There are zero deployed applications using this system. No persisted data exists. Any format, ID, or behavior can change freely.

## Problem

The current TypeMapper requires a unique GUID registration for every closed generic instantiation. `IList<MyTypeA>`, `IList<MyTypeB>`, `IList<MyTypeC>` each need their own pre-assigned GUID in the generated `AutoGeneratedCurrentAssemblyTypeMapperForRefactoringSupport.cs` files.

This causes a geometric explosion: with N generic definitions and M type arguments, the number of required registrations grows as N × M^k (where k is the arity). Adding a new type argument to a project forces regeneration of mapping files with new entries for every generic that uses it.

## Solution

Compute TypeIds for closed generics deterministically from the TypeIds of their components. Only leaf types and open generic definitions need explicit GUID registrations.

### How It Works

All composition uses a single namespace GUID (`GenericCompositionNamespace`) and a single rule: hash the "definition" TypeId + ordered type argument TypeIds.

1. **Leaf types** and **open generic definitions** get explicit GUIDs in generated files (as today)
2. **Closed generics** get computed GUIDs: `DeterministicGuid(GenericCompositionNamespace, TypeId(IList<>).Bytes ++ TypeId(MyType).Bytes)`
3. **Arrays** use the same mechanism with a synthetic `ArrayMarkerTypeId` constant (since .NET arrays are not generic types — `typeof(int[]).IsGenericType` is `false` and there is no `Array<T>` open generic definition): `DeterministicGuid(GenericCompositionNamespace, ArrayMarkerTypeId.Bytes ++ TypeId(MyType).Bytes)`
4. **Nested generics** compose recursively (bottom-up): `IDict<IList<MyType>>` → compute `TypeId(IList<MyType>)` first, then use that result as a type argument to compute `TypeId(IDict<IList<MyType>>)`

### Deterministic GUID Computation (UUID v5, RFC 4122)

The public API takes the definition TypeId and type argument TypeIds — nothing else:

```csharp
static TypeId ComputeCompositeTypeId(TypeId definitionTypeId, params TypeId[] typeArgumentTypeIds)
```

Internally this uses UUID v5: SHA-1 hash of a fixed namespace constant + the concatenated GUID bytes of all components, formatted as a standard GUID with version/variant bits set. The namespace constant is a hardcoded implementation detail — callers never see it.

#### Examples

Simple generic:
```
TypeId(IList<MyType>) = ComputeCompositeTypeId(TypeId(IList<>), TypeId(MyType))
```

Multi-param generic:
```
TypeId(IDict<string, int>) = ComputeCompositeTypeId(TypeId(IDict<,>), TypeId(string), TypeId(int))
```

Array:
```
TypeId(MyType[]) = ComputeCompositeTypeId(ArrayMarkerTypeId, TypeId(MyType))
```

#### Nested Generics

Computation is bottom-up. Each level produces a single GUID before being used as input to the next:

```
// Step 1: IList<MyType> → single GUID
var innerTypeId = ComputeCompositeTypeId(TypeId(IList<>), TypeId(MyType))
// innerTypeId is now just a Guid, e.g. "7a3f..."

// Step 2: IDict<string, IList<MyType>> → uses that GUID as an argument
var outerTypeId = ComputeCompositeTypeId(TypeId(IDict<,>), TypeId(string), innerTypeId)
```

No ambiguity — by the time a nested generic is used as a type argument, it's already been reduced to a single GUID. The payload for step 2 is exactly three 16-byte GUIDs concatenated, regardless of how `innerTypeId` was computed.
```

### Collision Risk

UUID v5 has 122 usable bits (6 reserved for version/variant). Birthday bound formula: probability ≈ n² / (2 × 2¹²²).

| Total TypeIds | Collision probability | Context |
|---|---|---|
| 1,000 | ~4.7 × 10⁻³² | Small app |
| 10,000 | ~4.7 × 10⁻³⁰ | Large app |
| 100,000 | ~4.7 × 10⁻²⁶ | Very large enterprise system |
| 1,000,000 | ~4.7 × 10⁻²² | A million distinct types (extreme) |
| 1,000,000,000 | ~4.7 × 10⁻¹⁶ | A billion types (absurd) |

Even at a billion types, the probability of an undetected ECC RAM bit flip (~10⁻¹² to 10⁻¹⁵ per year) is orders of magnitude higher. Compze is a generic framework with unbounded consumer type counts, but no realistic application approaches even 100,000 TypeIds.

Deterministic GUIDs occupy the same 122-bit space as `Guid.NewGuid()`, which the system already trusts. The risk profile is identical — values are chosen via SHA-1 instead of CSPRNG.

## Runtime Flow

No changes to `ITypeMapper` or downstream consumers — TypeId remains a `Guid`.

1. **Assembly loads** → existing `TypeMapperTypeDiscovery` scan finds all closed generic instantiations (unchanged)
2. **For each discovered closed generic**, instead of looking up a pre-assigned GUID, compute it deterministically from the registered component GUIDs
3. **Register** the computed TypeId in both `TypeToTypeIdMap` and `TypeIdToTypeMap` (same as today)
4. **`GetId`/`GetType`** remain simple dictionary lookups

## What Changes

- **Generated mapping files** shrink to only leaf types + open generic definitions. No entries for closed generics or arrays.
- **`TypeMapperTypeDiscovery.ShouldMapType`** changes to also accept `IsGenericTypeDefinition` (open generics need registration)
- **`TypeMapperTypeDiscovery.CollectClosedGenericTypes`** still runs to discover which combinations exist, but its output feeds deterministic computation instead of file lookup
- **`TypeMapperSourceCodeGenerator`** stops emitting entries for closed generics and arrays
