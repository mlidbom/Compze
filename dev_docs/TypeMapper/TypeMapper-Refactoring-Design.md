# TypeMapper Refactoring Design

> **Draft.** This is a direction, not a specification. Adjust as we go — the goal is clean separation of concerns, not rigid adherence to this exact shape.

## Problem

TypeMapper is a god object. It scans assemblies, reads mapping files, validates mappings, computes deterministic IDs, generates error messages with source code, writes files to disk, and serves as the runtime lookup dictionary — all in one class with interleaved static methods and shared mutable state.

## Design Direction

### Type Hierarchy: Wrapping `System.Type`

Replace the constant `IsGenericTypeDefinition` / `IsArray` / `IsGenericType && !IsGenericTypeDefinition` checks with a hierarchy that makes the domain distinctions explicit:

```
TypeMapperType
├── ExplicitlyMappedType        — needs a hand-assigned GUID
│   ├── LeafType                — non-generic, non-array concrete or abstract type
│   └── OpenGenericDefinition   — e.g. List<>, Dictionary<,>, Taggregate<,,,,>
└── ComputedTypeIdType              — TypeId computed deterministically from components
    ├── ClosedGenericType       — e.g. List<MyEntity>, has OpenGenericDefinition + type arguments
    └── ArrayType               — e.g. MyEntity[], has element type
```

Each wraps a `System.Type` and exposes only what type-mapping needs. Construction enforces the classification — you can't accidentally treat a closed generic as a leaf type.

`ComputedTypeIdType` knows its components (the `ExplicitlyMappedType` or other `ComputedTypeIdType` instances it's built from). A `ClosedGenericType` holds its `OpenGenericDefinition` plus its type arguments as `TypeMapperType[]`. An `ArrayType` holds its element type as a `TypeMapperType`. This makes the recursive structure explicit.

### Assembly Scanner

`TypeMapperAssemblyScanner` takes an `Assembly` and returns an `AssemblyTypeMap` (or similar) containing the collections:

- `LeafTypes` `OpenGenericDefinitions`, `ClosedGenericTypes`, `ArrayTypes` found in that assembly

Classification is purely structural — determined by `System.Type` properties alone (`IsArray`, `IsGenericTypeDefinition`, etc.). No TypeIds, no mapping data. A `LeafType` is always an `ExplicitlyMappedType` because of what it *is*, not because of what's been registered.

Pure function: assembly in, classified types out. No side effects, no shared state.

### Mapping File Reader

A dedicated class reads the `AutoGeneratedCurrentAssemblyTypeMapperForRefactoringSupport` class from an assembly and returns the existing `Type → TypeId` pairs. Pure data extraction — no validation, no error messages, no wrapping in the type hierarchy.

### Mapping Correlator

Takes the scanner output (`AssemblyTypeMap`) and the mapping reader output (existing `Type → TypeId` pairs), correlates them, and produces a result describing:

- Which `ExplicitlyMappedType` instances have TypeId assignments (matched)
- Which `ExplicitlyMappedType` instances are missing assignments (gaps)
- Which `ComputedTypeIdType` instances can have their TypeIds computed (all components resolved)
- Which `ComputedTypeIdType` instances are blocked (missing component mappings)

This result is a custom type with clear semantics — not a raw dictionary.

### Deterministic ID Computation

`DeterministicTypeIdGenerator` (already exists) operates on `ComputedTypeIdType` instances. Given that each component's TypeId is resolved, it computes the composite TypeId. The recursion is natural: a `ClosedGenericType` asks for its `OpenGenericDefinition`'s TypeId and each argument's TypeId, then composes them.

### Missing Mapping Reporter

Takes the correlator's output describing gaps and generates the error messages / source code snippets. Separate from the detection logic. Could also handle the (currently disabled) file-writing when we re-enable that.

### TypeMapper Itself

Becomes a thin coordinator:

- Owns the `MappingState` (the running `Type ↔ TypeId` dictionaries)
- On assembly load: delegates to scanner → reader → correlator → computation, then merges results into state
- `GetId` / `GetType` / `TryGetType`: dictionary lookups (unchanged)
- `GetIdForTypesAssignableTo`: lookup + filter (unchanged)
- Error paths: delegates to the reporter

The assembly-load handler and `ReentrancyGuard` stay here since they're about coordinating the process, not doing the work.

## Key Principles

- **Each class has one job.** If you can't name it in 3 words, it's doing too much.
- **Immutable intermediate results.** Scanner output, reader output, correlator output — all immutable data. Only `MappingState` (owned by TypeMapper) mutates.
- **The type hierarchy encodes domain knowledge.** The fact that a closed generic's TypeId depends on its definition and arguments isn't implicit in conditional branches — it's explicit in the class structure.
- **Testable in isolation.** Scanner can be tested with a test assembly. Reader with a test mapping class. Correlator with hand-built inputs. Generator with known TypeIds. None requires the full TypeMapper to exercise.
