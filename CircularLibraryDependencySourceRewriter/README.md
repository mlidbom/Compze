# CircularLibraryDependencySourceRewriter

A tool that resolves circular dependencies between .NET library projects by internalizing source code. It copies C# source files, rewrites all `public` type declarations to `internal`, and prepends an `<auto-generated>` header — allowing one library to include a private copy of another library's code without creating a NuGet package dependency cycle.

## The Problem

You have two library projects where A depends on B, but B also needs some code from A:

```
A ──references──▶ B       (normal project reference)
B ──needs code from──▶ A  (would create a circular dependency)
```

You can't add a reverse project reference from B to A. Merging them sacrifices modularity. Extracting a third shared project isn't possible when the needed code is tightly coupled to A.

## The Solution

The library that **cannot** add the project reference (B) internalizes a copy of the code it needs from the other library (A). The types are rewritten to `internal`, so:

- They don't conflict with the originals at the API level
- They can't leak across assembly boundaries
- Consumers of both libraries see no duplicates

```
A  ──references──▶ B                              (normal project reference)
B  (contains internal copy of needed code from A)  (no reverse dependency needed)
```

The other side (A) simply uses a normal project reference to B — no internalization needed.

## What Gets Rewritten

| Input | Output |
|-------|--------|
| `public class Foo` | `internal class Foo` |
| `public interface IBar` | `internal interface IBar` |
| `public struct Baz` | `internal struct Baz` |
| `public enum Color` | `internal enum Color` |
| `public record Person` | `internal record Person` |
| `public delegate void Handler` | `internal delegate void Handler` |
| `public static class Extensions` | `internal static class Extensions` |
| `public abstract class Base` | `internal abstract class Base` |
| `public sealed partial class Widget` | `internal sealed partial class Widget` |

Every output file is prepended with:
```csharp
// <auto-generated>
// This file was generated by CircularLibraryDependencySourceRewriter.
// Changes to this file will be overwritten.
// </auto-generated>
```

### MSBuild Task: `RewriteDirectoryTask`

An MSBuild `ITask` with two required properties:
- `InputDirectory` — path to the source files to internalize
- `OutputDirectory` — path where rewritten files are written

## MSBuild Integration

Import the `.targets` file in the consuming project and set two properties:

```xml
<Import Project="path\to\CircularLibraryDependencySourceRewriter.targets" />

<PropertyGroup>
  <InternalizeSourceFrom>..\OtherLibrary</InternalizeSourceFrom>
  <InternalizeSourceTo>$(MSBuildProjectDirectory)\InternalizedSource</InternalizeSourceTo>
</PropertyGroup>
```

The target:
- Runs `BeforeTargets="CoreCompile"` so rewriting happens before your code compiles
- Uses MSBuild `Inputs`/`Outputs` for incremental builds — skips if no source files changed
- Auto-includes the generated files in compilation
- Only runs when both properties are set

The `InternalizedSource` folder can be `.gitignore`d since it's regenerated on build.

## API

### `SourceRewriter.MakeTypesInternal(string source) → string`

Rewrites a single C# source string. Prepends the auto-generated header and replaces `public` type declarations with `internal`.

### `SourceRewriter.RewriteDirectory(string inputDirectory, string outputDirectory)`

Rewrites all `.cs` files from `inputDirectory` into `outputDirectory`, preserving directory structure. Deletes `outputDirectory` first if it exists.
