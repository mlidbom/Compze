# Compze.InternalizedSourceReferences

A tool that resolves circular dependencies between .NET library projects by internalizing source code. It copies C# source files, rewrites all `public` type declarations to `internal`, and prepends an `<auto-generated>` header — allowing one library to include a private copy of another library's code without creating a NuGet package dependency cycle.

Distributed as a NuGet package containing an MSBuild task that runs automatically during build.

## The Problem

You have two library projects where A depends on B, but B also needs some code from A:

```
A ──references──▶ B       (normal project reference)
B ──needs code from──▶ A  (would create a circular dependency)
```

You can't add a reverse project reference from B to A. Merging them sacrifices modularity. Extracting a third shared project isn't always feasible.

## The Solution

The library that **cannot** add the project reference (B) internalizes a copy of the code it needs from the other library (A). The types are rewritten to `internal`, so:

- They don't conflict with the originals at the API level
- They can't leak across assembly boundaries
- Consumers of both libraries see no duplicates

```
A  ──references──▶ B                              (normal project reference)
B  (contains internalized version of A's code)
```

### Key Constraints and gotchas

**Internalized types cannot appear in the public API** of the consuming project. Since they are rewritten to `internal`, they can't be used in public method signatures, property types, base classes, or interface implementations that are exposed to consumers. If a type needs to be in a public API, it belongs in a shared lower-level project with a normal `ProjectReference`.

**Static state** Any static state in the source interned library will NOT be shared. The interned source results in entirely different types and data as far as the runtime is concerned.
 

### Incremental Writes

The rewriter only writes output files when their content has actually changed. This prevents build tools (like NCrunch) from detecting false changes and triggering unnecessary rebuild loops.

## What Gets Rewritten

| Input | Output |
|-------|--------|
| `public class Foo` | `internal class Foo` |
| `public interface IBar` | `internal interface IBar` |
| `public struct Baz` | `internal struct Baz` |
| `public enum Color` | `internal enum Color` |
| `public record Person` | `internal record Person` |
| `public delegate void Handler` | `internal delegate void Handler` |
| `public static class Extensions` | `internal static class Extensions` |
| `public abstract class Base` | `internal abstract class Base` |
| `public sealed partial class Widget` | `internal sealed partial class Widget` |

Every output file is prepended with:
```csharp
// <auto-generated>
// This file was generated by Compze.InternalizedSourceReferences.
// Changes to this file will be overwritten.
// </auto-generated>
```

## MSBuild Integration

### NuGet Package

Add a `PackageReference` to the consuming project:

```xml
<PackageReference Include="Compze.InternalizedSourceReferences" Version="0.1.0-local" PrivateAssets="all" />
```

The `.targets` file is auto-imported from the package. Set two properties to enable rewriting:

```xml
<PropertyGroup>
  <InternalizeSourceFrom>..\OtherLibrary</InternalizeSourceFrom>
  <InternalizeSourceTo>$(MSBuildProjectDirectory)\InternalizedSource</InternalizeSourceTo>
</PropertyGroup>
```

To internalize from multiple source directories, separate them with semicolons:

```xml
<PropertyGroup>
  <InternalizeSourceFrom>..\LibraryA;..\LibraryB;..\LibraryC</InternalizeSourceFrom>
  <InternalizeSourceTo>$(MSBuildProjectDirectory)\InternalizedSource</InternalizeSourceTo>
</PropertyGroup>
```

The target:
- Runs `BeforeTargets="CoreCompile"` so rewriting happens before your code compiles
- Only writes files when content has actually changed (avoids triggering rebuild loops)
- Removes stale output files that no longer have a corresponding input
- Auto-includes the generated files in compilation
- Only runs when both properties are set

The `InternalizedSource` folder can be `.gitignore`d since it's regenerated on build.

## API

### `SourceRewriter.MakeTypesInternal(string source) → string`

Rewrites a single C# source string. Prepends the auto-generated header and replaces `public` type declarations with `internal`.

### `SourceRewriter.RewriteDirectory(string inputDirectory, string outputDirectory)`

Convenience overload — calls `RewriteDirectories` with a single input directory.

### `SourceRewriter.RewriteDirectories(string[] inputDirectories, string outputDirectory)`

Rewrites all `.cs` files from multiple input directories into `outputDirectory`, preserving directory structure per input. Only writes files whose content has actually changed. Removes stale output files that no longer have a corresponding input.
