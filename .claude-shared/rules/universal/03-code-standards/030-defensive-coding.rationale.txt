# Rationale — `03-defensive-coding.md`

Read before weakening this rule.

## Why "never preemptively swallow exceptions" is stated so absolutely

We've observed time and time again how coders, both human and agents, "fix" things this way and it causes nothing 
but trouble over the long run.


Moreover, this was tested on a sibling codebase. Every `catch (Exception)` was deleted. Across real use, exactly one
removal surfaced a crash — and that catch had been *masking a real wrong-desktop bug*, not handling a
legitimate non-bug failure. Removing it didn't cause the failure; it revealed a latent one we then fixed
properly. Net result: zero of the catches were motivated, and the single "regression" was the rule working
as designed.

So the rule is **empirically grounded, not aspirational.** If a future you is tempted to add a "just in
case" catch — or to soften the rule to permit them — this is the evidence that says don't: delete the catch
and let it crash until you have your own empirical evidence of a specific, characterized non-bug exception.
