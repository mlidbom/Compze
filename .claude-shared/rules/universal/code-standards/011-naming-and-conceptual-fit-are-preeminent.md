# Naming and conceptual fit are preeminent; implementation detail is a distant also-ran

Clear, intuitive mental models — names that match their abstraction, concepts that cohere — are the thing the user values most in code. Implementation details rank far below. Default to that ranking in everything: design discussion, review, what you choose to raise.

- When the user remarks on code without explicitly flagging a **bug**, assume 90%+ they mean questionable **design** in this sense — a name that lies, an abstraction that doesn't cohere, a muddled mental model. Read the remark there FIRST, not as an implementation detail.
- A name that contradicts how the thing is used is a **modeling error to fix**, never a detail to explain away. Never rebut a naming/conceptual critique by citing how the implementation happens to work ("it's used as a value, so the name is fine") — that talks past the user and is a constant, major source of misunderstanding.
- Engage where the user raised it: does the name/abstraction cohere with the concept it evokes? If not, propose renaming or re-modeling so the two match, and let them pick the direction.
- **Conceptual fit decides the answer; it is not a factor to weigh against pragmatics.** Usage, performance, YAGNI, "how the implementation happens to work" inform only *how* to reach the coherent model — never *whether* it is right. Catching yourself listing conceptual fit as one consideration in a balanced trade-off means you have already lost the thread: lead with the coherent re-modeling, then note its cost. Never the reverse; never recommend against the coherent model on pragmatic grounds. ("Is this the right *kind* of type — value vs entity, class vs struct?" is one of these questions, not a detail.)
- **The user's "clean / simple / maintainable" are defined terms, not plain English.** They mean conceptual coherence and relentless extraction-for-clarity — often the opposite of the median reading (fewer parts, don't-abstract-until-reused). Do not collapse them to what they usually mean; read them as the standard in [conceptual coherence](010-conceptual-coherence-and-naming.md).
