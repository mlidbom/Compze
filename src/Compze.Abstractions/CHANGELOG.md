# Changelog

All notable changes to Compze.Abstractions will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/).

## 0.2.0-alpha

- **The tessaging content leaves: this package is the identity, time and serialization primitives now.** The tessage type system (the `ITessage` hierarchy, the `TessageTypes` families, `TessageId`, the publisher-identifying tevent wrapper, the publisher/sender door contracts, and the tessage validation) moved to the new `Compze.Tessaging.Abstractions`. The endpoint contracts — `IEndpoint`, `IEndpointHost`, `EndpointId`, `EndpointConfiguration`, `ReadinessTypes`, `EndpointNotReadyWithinPatienceException`, and the discovery trio `EndpointAddress`/`IEndpointRegistry`/`IEndpointAddressAnnouncer` — moved into `Compze.Tessaging` itself (`Compze.Tessaging.Endpoints` and `Compze.Tessaging.Endpoints.Discovery`), as did `ITessagingSerializer` and `ITypermediaSerializer`. An endpoint is a tessaging concept — the immediate-consistency boundary of the consistency law — so `Compze.Abstractions` was never its truthful home. What remains here is what the rest of the framework genuinely shares: `EntityId` and its branches, the `Entity<>` base classes, `ValueWrapper<>`, `IUtcTimeTimeSource`, `IJsonSerializer`/`IDocumentDbSerializer`, and `IConfigurationParameterProvider`.

- **`IEndpoint` is the first-class endpoint contract: `StartAsync` joins it, the six phase methods leave it.** An endpoint is constructed, started (`StartAsync` drives its own phase ordering — listen → announce → send), and disposed (disposal drives the mirror) with no host anywhere; `IEndpointHost` is re-described as what it is — an optional convenience owning several endpoints' lifecycles in one process, adding nothing an endpoint cannot do alone, with no ordering between the endpoints. The individual phase methods remain available on the concrete `Endpoint` in Compze.Tessaging for callers that script a phase deliberately.
- **Readiness joins the endpoint contract: `IEndpoint.AwaitReadinessAsync(ReadinessTypes, patience?)`.** Completes when the endpoint can reach a handler for every named type; exhausted patience throws the new `EndpointNotReadyWithinPatienceException`, naming every type still unavailable and what the endpoint's peer memory remembers about each. The type sets are `ReadinessTypes`: reflection factories (`InAssemblyContaining`, `InNamespaceOf` with levels-to-walk-up, and the explicit `These`) admitting only the remotable single-handler tessage kinds — tueries, typermedia tommands, exactly-once tommands — with anything else, or a set that would be empty, failing loud at composition instead of as a timeout later.
- **Synchrony follows the type at the tessaging doors.** `IUnitOfWorkTeventPublisher` and `IIndependentTeventPublisher` become sync/async pairs: `PublishAsync` serves every tevent kind, awaiting participation and the durable outbox write an `IExactlyOnceTevent`'s contract demands, while the synchronous `Publish` serves the kinds whose contract keeps sync first-class — strictly-local and best-effort — and refuses an exactly-once tevent loudly, pointing at `PublishAsync`. `IUnitOfWorkTommandSender` and `IIndependentTommandSender` go async-only (`void Send` dies, `Task SendAsync` is the door): `IExactlyOnceTommand` is statically exactly-once, and exactly-once kinds are async end to end.
- The publish and send front doors are named by the caller's relationship to the unit of work, and both flavors exist. `ITeventPublisher` is `IUnitOfWorkTeventPublisher` (publishes within the caller's unit of work) and gains its independent counterpart `IIndependentTeventPublisher`: a root-resolvable singleton for code outside any unit of work — each publish runs as its own, committed when the call returns. `IServiceBusSession` is `IUnitOfWorkTommandSender` — it held no conversational state, so it was never a session, and its summary over-claimed ("dispatches tessages between processes"; it sends exactly-once tommands) — paired with the new `IIndependentTommandSender`. The independent doors assert their independence: called from within an ambient transaction they throw instead of silently joining it.

- `IEndpointRegistryAndAnnouncer` — a registry with both faces of endpoint discovery: the `IEndpointRegistry` read side and the `IEndpointAddressAnnouncer` write side — what an endpoint participates in (`ParticipateIn(...)`). `ITestingEndpointHost.EndpointRegistry` exposes the testing host's own such registry, which every endpoint in the host participates in.
- `IEndpointRegistry.AwaitPossibleMembershipChange(timeout, cancellationToken)`: how a router waits between reconciliation passes. A registry that can observe its own changes wakes its waiters at signal latency, so topology changes propagate nearly instantly; the default implementation waits out the timeout — right for a registry whose membership never changes on its own initiative. The timeout is not merely a fallback: no signal can announce a crashed process, and a failed connection needs a retry no membership change will trigger, so the periodic pass it drives must exist regardless.
- Declaring no `IEndpointRegistry` now means the endpoint discovers nothing and only serves that communication style. The configuration fallback (`AppConfigEndpointRegistry`, a never-implemented placeholder that crashed the endpoint at start) is deleted. Nothing self-addressed crosses the wire: a tommand whose handler is in the endpoint's own roster executes inline, in the sender's execution (the consistency law), so an endpoint's conversation with itself needs no discovery.
- `IUnitOfWorkTeventPublisher` — the one way to publish a tevent, for anything that publishes (the tevent store forwarding committed taggregate tevents is just its most common client). It routes each tevent by the delivery contract the tevent's type declares: synchronous in-process delivery within the caller's transaction always (plus observation, outside it); durable remote delivery for an `IExactlyOnceTevent` and best-effort remote delivery for any other `IRemotableTevent` when the endpoint's composition wires the legs. Publishing requires the caller's ambient transaction — with none present it throws, and code outside any unit of work publishes through `IIndependentTeventPublisher`. See `src/Compze.Tessaging/dev_docs/tevent-delivery-model.md`.
- The hosting contracts are the shape endpoints compose into, not a plug-in seam: `IEndpoint` is what a composed endpoint is (its container's `ServiceLocator`, `IsRunning`, and the six lifecycle-phase methods — listen → announce → send up, retract → stop sending → stop listening down), and `IEndpointHost.RegisterEndpoint<TEndpoint>(Func<IContainerBuilder, TEndpoint>)` is the one registration door: the host hands out a fresh container builder and owns the composed endpoint's lifecycle, never knowing its tier. `IEndpointBuilder` (with `GetOrAddFeature`/`OnContainerBuilt`/`AddComponent`), `IEndpointComponent`, the typed foundation (`ComposeEndpoint`/`EndpointComposer`/`EndpointFoundation`), the serializer-slot interfaces (`ITessagingSerializerSlot`, `ITypermediaSerializerSlot`), and `ITestingEndpointHost` are gone with the feature machinery they served — composition choices are parameters on the concrete endpoint builders in `Compze.Tessaging` now.
- Announcement is its own host-wide lifecycle phase — listening, then announcing (`AnnounceAddressAsync`), then sending, with retraction (`RetractAddressAsync`) mirroring on stop before any sending stops. Every endpoint's listening completes before any announcement, and every announcement before any sending — so an announced address is always one that is actually listening, and a router's first look at an `IEndpointRegistry` when sending starts sees every endpoint its host announced, never a partial membership decided by start-up racing.
- `IEndpointAddressAnnouncer`: the announcement timing contract is host-ready rather than endpoint-local — the announced address is the endpoint's one transport-server address, serving every distributed capability the endpoint speaks, announced in the host's announcing phase (above).
- Added the `Tevents()` projection: a sequence of `IPublisherIdentifyingTevent<TTevent>` wrappers projects to the wrapped inner tevents.
- The delivery-tier interfaces now document their settled contracts: `IAtMostOnceTessage` (kept for best-effort send + receiver dedup — the UI double-click case) and `IExactlyOnceTessage` (the durable tier) point at the tevent delivery model (`src/Compze.Tessaging/dev_docs/tevent-delivery-model.md`), replacing the stale open-question comments that document answered.
- The publisher-identifying wrapper carries only publisher identity, not delivery-guarantee markers: `IPublisherIdentifyingTevent<out TTevent>` is the sole wrapper interface, and it has a type-identifier mapping so closed generics over it can travel in persisted and transmitted type references. A tevent's delivery guarantee lives on the tevent itself and is read via covariance (`IPublisherIdentifyingTevent<IExactlyOnceTevent>`); the earlier `IRemotablePublisherIdentifyingTevent<>`/`IExactlyOncePublisherIdentifyingTevent<>` tiers, which re-declared the inner's guarantee interfaces onto the wrapper, are removed.
