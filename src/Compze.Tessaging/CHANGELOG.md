# Changelog

All notable changes to Compze.Tessaging will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/).

## 0.4.0-alpha

- **Endpoints are first-class: `Endpoint.StartAsync` drives the endpoint's own lifecycle.** `StartAsync` runs the up-phases in order — listen → announce → send — and disposal already drove the mirror, so an endpoint is composed, started, and disposed with no host anywhere; an `EndpointHost` is an optional convenience starting and disposing several together, with no ordering between them (see the Compze.Hosting changelog). The six phase methods stay public on `Endpoint` for callers that script a phase deliberately — a specification stopping the listening to observe delivery waiting out downtime, a late-started endpoint. An endpoint announces its address in its own announcing phase — after its listening phase, before its sending phase — so an announced address is always one that is actually listening, in every process topology.
- **Storage belongs to the domain — an endpoint joins a domain database.** The foundation declaration names the domain database this endpoint joins (`ExactlyOnceEndpointBuilder.DomainDatabase`, filled by a database package's named extension — `SqliteDomainDatabase(...)` and kin): the database is the domain's, never the endpoint's — the domain data the endpoint's executions touch lives there, and exactly-once atomicity *is* the co-location. Each endpoint owns a **prefixed table-set** in it (`EndpointTableSet`: its inbox, its outbox and outbox dispatching, and its durable peer memory, each table prefixed `«EndpointName»_` — `Backend_InboxTessages`), which is what lets any number of endpoints store side by side in one domain database and converse exactly-once through prefixed outboxes and inboxes. The prefix makes an exactly-once endpoint's name identifier material — a letter followed by letters, digits, or underscores, at most 28 characters (PostgreSQL's 63-byte identifier limit minus the longest generated identifier) — asserted loud at composition, never sanitized silently. The domain-level tables (the type-id interner, the tevent store, the document db, the endpoint catalog) are deliberately unprefixed: they are the domain's data, shared by every endpoint that joins — on sqlite this also means co-located endpoints share one interner database, since its name derives from the domain database's.
- **The endpoint catalog and the process lease: one process per endpoint, enforced.** Every domain database carries one shared `EndpointCatalog` table — each endpoint's name, `EndpointId`, creation time, and process lease. It enforces name uniqueness (a name only ever belongs to one endpoint, and an id never silently re-keys itself under a new name — renaming means decommissioning the old storage; both fail loud at startup) and the one-process-per-endpoint rule: the process lease is a heartbeat lease (`EndpointProcessLease`), claimed as the first act of starting to listen — whether this process may run the endpoint is decided before anything else touches the database, and a refused claim throws with the endpoint still fully un-started. A claimant finding the lease held waits out one lease duration (`ExactlyOnceEndpointBuilder.ProcessLeaseDuration`, default 15 seconds; heartbeats at a fifth of it) — a crashed predecessor's lease goes stale within that and is taken over silently, so crash recovery needs no manual cleanup — and only a holder proven alive by its heartbeats fails the start loud (`EndpointAlreadyRunningInAnotherProcessException`, naming the holder). A live holder whose lease is stolen out from under it (a debugger pause, a machine sleep) reports through the background-exception machinery. The lease is released at disposal, after the observation drain, once nothing in the process writes to the domain database; every claim is one conditional SQL statement, so racing claimants serialize on the database, and every catalog act suppresses the ambient transaction — the lease is its own act.
- **Waiting sends: a send with no live route waits, bounded, instead of exploding.** Every remote-facing request/response send — a typermedia tuery or tommand navigated remotely, an exactly-once tommand whose receiver binding has no bindable receiver — now waits, within the endpoint's **handler-availability patience** (a flat 30 seconds unless the composition declares otherwise: `EndpointBuilder.HandlerAvailabilityPatience`), for the world to become right — a first contact, a known peer's return, an ambiguity resolving — then proceeds normally. Only exhausted patience fails loud, and the no-handler exception family (`NoHandlerForTypermediaTypeException`, `MultipleHandlersForTypermediaTypeException`, `NoHandlerForTessageTypeException`, `MultipleHandlersForTessageTypeException` — all public now, since they reach the sending application code) is exclusively the patience-exhausted failure: the message says what was waited for, for how long, and what the peer memory remembers — known-but-down names the remembered peer, never-seen names the probable deployment or configuration error. The exactly-once wait strictly precedes the one bind-at-send, so the exactly-once in-order guarantee is untouched; with several remembered handlers and none live, the send binds to the one that connects — live is current by definition. The external client's typermedia router keeps its immediate throws: its connections change only by its own explicit connects, so there is no discovery to wait for.
- **Readiness: `IEndpoint.AwaitReadinessAsync(ReadinessTypes, patience?)`.** Completes when the endpoint can reach a handler for every named type — its own roster serves it, an exactly-once tommand type has a bindable receiver, or a request/response type has exactly one live route: precisely the availability a send would not have to wait for, so readiness and waiting sends compose instead of overlapping. The framework-native replacement for hand-rolled startup readiness probes: awaited before opening traffic it front-loads the discovery wait, an orchestrator's readiness probe wires to it, and exhausted patience throws `EndpointNotReadyWithinPatienceException` naming every type still unavailable and what the peer memory remembers about each.
- **Peer memory answers the single-handler question for every remotable single-handler kind.** `IPeerRegistry.HandlerIdsFor` is Type-keyed now — the exact match over every advertised tuery, typermedia tommand, and exactly-once tommand — serving both the send-time receiver binding and the known-but-down vs never-seen distinction.
- **Synchrony follows the type, end to end.** The tevent publishers become sync/async pairs (`Publish`/`PublishAsync`, on the unit-of-work and independent doors alike): `PublishAsync` serves every tevent kind, awaiting participation and the durable outbox write an `IExactlyOnceTevent`'s contract demands, while the synchronous `Publish` serves the kinds whose contract keeps sync first-class — strictly-local and best-effort — and refuses an exactly-once tevent loudly, pointing at `PublishAsync` (C# cannot exclude a subtype statically, so the type contract is enforced at the door, exactly as the handler declaration verbs enforce theirs). The tommand senders go async-only — `Task SendAsync(IExactlyOnceTommand)`, the inline in-roster execution awaited — because `IExactlyOnceTommand` is statically exactly-once. Administration is async (`IPeerAdministration.DecommissionAsync`), and the whole pipeline beneath is genuinely async: the `ITessagingSqlLayer` members and all four backend implementations, the outbox and inbox storages, the peer vertical (`IPeerRegistry.RecordAdvertisementAsync` in the router's connect flow, the lifecycle observers, the decommission participants), the inbox's handler execution (transaction envelope, storage marks, retry loop), the best-effort direct dispatch, and the typermedia handler executor serving the wire — with the ambient transaction flowing across every await (`ExecuteUnitOfWorkAsync`), and commit hooks bound to the transaction captured before the first await. The strictly-local navigators deliberately keep sync first-class, and the remote navigator was async-first already. The one deliberate sync-context bridge is the tevent store's, where the Teventive taggregate model raises tevents synchronously; the exactly-once delivery stream remains a deliberately synchronous dedicated-thread head-of-line pump. Session affinity became transactional in the same stroke — the thread-affinity usage guard died with the thread-pinned era (see the Compze.Internals.SystemCE changelog).
- **Observation redesigned: committed facts only, off-thread, per-observer FIFO.** An observer observes facts, and an uncommitted tevent is not yet a fact: a locally published tevent is queued for its observers when its publishing unit of work commits (`IUnitOfWorkResolver.OnCommittedSuccessfully`) — a rolled-back publish is never observed, inverting the previous fires-immediately-even-for-doomed-transactions semantics — while an arriving tevent queues on arrival, already a committed fact on its publisher. Dispatch runs off-thread on the engine's `TeventObservationDispatcher` (the name returns from the pre-engine per-path choreography, now naming the engine's one background machinery): an observer's failure never fails the caller (it routes to the background-exception reporter), an observer's latency never blocks it, and each observer's own FIFO queue keeps an observing read model seeing tevents in publish/arrival order — a tevent a participation handler publishes in response to another is observed after its cause. The dispatch pump starts with `ExecutionContext` flow suppressed, so the committing caller's ambient transaction cannot leak into observer scopes. Nothing escapes the tests or teardown: the dispatcher reports queued/dispatched transitions to the tessages-in-flight tracker, so the testing host's at-rest wait covers the observation queues (with pending observations listed in the timeout diagnostics), and an endpoint drains the dispatcher at disposal — after listening stops, while the container still serves the observers their scopes. The `ITessagesInFlightTracker` seam moves to `Compze.Tessaging.Implementation.Abstractions` (implementations to `Compze.Tessaging.Implementation`): covering observation, it stopped being a transport-only device.
- **The concrete endpoint types** (`Compze.Tessaging.Endpoints`): an endpoint is an engine given identity and a wire, and there are exactly two endpoint types, differing only in what crossing the endpoint boundary guarantees — `BestEffortEndpoint` (no database; process-lifetime peer memory, per-peer best-effort tevent queues) and `ExactlyOnceEndpoint` (everything above, plus the durable vertical: inbox, outbox, durable peer memory, and the tommand-sending doors). Both serve all four tessage kinds, unconditionally, and each is a plain composition root driving its own lifecycle phases in readable methods. Composition choices are parameters, not plugins: `Compose(container, name, id, endpoint => ...)` hands the callback the tier's declaration surface — the transport protocol, the endpoint's one serializer, the domain database the endpoint joins (exactly-once tier, whose named declaration registers the whole engine pairing), the topology declarations (`ParticipateIn`/`DiscoverEndpointsThrough`/`AnnounceAddressTo`, `RequirePeers`, `DoNotQueueTeventsFor`), and the same engine declaration block every composition speaks — and a missing declaration fails loud at composition, naming it. The whole endpoint feature machinery this replaces is deleted: the five endpoint features and their `Add*` extensions, `IEndpointBuilder` with `GetOrAddFeature`/`OnContainerBuilt`/`AddComponent`, the `IEndpointComponent` seam, the typed foundation (`ComposeEndpoint`/`EndpointFoundation`), the twin serializer slots, the per-style advertisement contributors (the advertisement has exactly one source now: the roster's one union projection), and the `TessagingAddress`/`TypermediaAddress` extension properties — the endpoint's one `Address` says it.
- **The pure client** (`TypermediaClient`): the third composed shape — a navigator and a transport client with no server, composed in its own container (`TypermediaClient.Compose`: the transport-client protocol, the one serializer parameter, the client's own type mappings), connecting to explicitly known addresses. It serves nothing, is discovered by nobody, and participates in no registry.
- **The consistency law's inline tommand execution**: the tommand-sender door (`IUnitOfWorkTommandSender`, and `IIndependentTommandSender` through it) consults the endpoint's `TessageHandlerRoster` — a tommand whose handler is in the roster executes inline, through the engine's one executor, in the sender's execution: exactly-once by construction (one transaction, no delivery machinery involved), immediate, and atomic with the sender — in-boundary failure fails the sender's execution, so everything rolls back together. Only a tommand whose handler lives elsewhere crosses the endpoint boundary, through the outbox. Because the roster is fixed per endpoint instance, a given tommand type is always-inline or never-inline for that endpoint — there is no per-send lottery. Nothing self-addressed ever crosses the wire anymore: the router's self-connection is deleted — reconciliation now excludes the endpoint's own announced address, routes lead only to *other* endpoints, and an answer claiming the endpoint's own identity fails loud (two processes under one `EndpointId` is a misconfiguration). The process-manager pattern — handling one tessage sends a follow-up tommand belonging to the same endpoint — is now in-boundary composition within the handling execution; its spec (`Given_an_exactly_once_tessaging_endpoint_declaring_no_discovery_registry`) is deliberately re-pinned from "delivered through its own inbox" to "executes inline in the sender's unit of work".
- **The LocalTessagingEngine** — the tessage-conversing heart of one container, declared in one composition block (`Compze.Tessaging.Engine`). The `TessageHandlerRoster` is the one immutable map for all four tessage kinds (replacing `TessageHandlerRegistry` and `TypermediaHandlerRegistry`), built from the declaration and closed: a second handler for a single-handler kind explodes at declaration, registration after the engine is built explodes, and every missing-handler lookup speaks one `NoHandlerException`. The `TessageHandlerExecutor` is the one implementation of handler execution, replacing the per-path choreography copies (`InProcessTeventPublisher`, `TeventObservationDispatcher`, the inbox's handler-execution switch, the direct dispatcher's and the typermedia executor's own envelopes) — a tevent's response is every compatible handler in the one unit of work, a tommand's its single handler in a unit of work, a tuery's its single handler in a scope.
- **One declaration surface everywhere**: `LocalTessagingEngine(engine => ...)` composes the engine into a plain container (replacing `InProcessTessaging()`/`InProcessTypermedia()`), and every endpoint feature and the endpoint builder speak the same verbs — `RegisterTessageHandlers(Action<TessageHandlerRegistrar>)` covering all four kinds (replacing `RegisterTessagingHandlers`/`RegisterTypermediaHandlers`/`RegisterHandlers`/`ForTommandWithResult` and the `ITessageHandlerRegistrar`/`ITypermediaHandlerRegistrar` interfaces) and `ObserveTevents(Action<TeventObservationRegistrar>)` for observation, under its own verb so the distinct semantics are visible at the declaration site (replacing `RegisterTransactionIgnoringTeventHandlers`). A registrar exists only inside its declaration callback; holding one and using it later explodes. Store integrations declare through the same surface: `RegisterTeventStore().HandleTaggregate` and `RegisterDocumentDb().HandleDocumentType` are handler contributors like any other.
- **Synchrony follows the type, enforced at declaration**: exactly-once kinds are async end to end — their handler verbs are async-only, and declaring a synchronous handler for an exactly-once tommand or an exactly-once-demanding tevent subscription explodes at declaration — while strictly-local and at-most-once kinds keep sync as first-class, with async available.
- The `Compze.Tessaging.Abstractions` package folds into `Compze.Tessaging`: the registrar interfaces (`ITessageHandlerRegistrar`, `ITransactionIgnoringTeventHandlerRegistrar` and their `CE` convenience extensions) now live in the `Compze.Tessaging.TessageHandling.Registration.Public` namespace. The separate package existed so a store could reference the registration surface without the Tessaging machinery, but no compiled code ever did — and the target design's direction is stores contributing to the engine, which the fold states in the project graph. The absorbed package's own history, preserved:
  - *0.3.0-alpha*: `ITessageHandlerRegistrar`'s `ForTevent` and `ForTommand` hand the handler an `IUnitOfWorkResolver`: every path that executes a tommand handler or delivers a participation tevent handler runs it inside a unit of work, and the signature now says so. Observation registration (`ITransactionIgnoringTeventHandlerRegistrar.ForTevent`) keeps `IScopeResolver` — observation is delivered detached from any transaction, deliberately.
  - *0.3.0-alpha*: `ITransactionIgnoringTeventHandlerRegistrar` — registers transaction-ignoring tevent handlers: observation, the one subscription-side opt-down from a tevent type's declared delivery guarantee. A separate registrar from `ITessageHandlerRegistrar`, deliberately: opting out of every delivery guarantee is visible and off the common surface.
  - *0.3.0-alpha*: `TessageHandlerRegistrarWithDependencyInjectionSupport` removed: the resolved-dependency `ForTevent`/`ForTommand` convenience overloads extend `ITessageHandlerRegistrar` itself (`TessageHandlerRegistrarCE`).
- `IPeerAdministration.Decommission(EndpointId)` — the decommission half of increment 6 of the durable peer topology (`src/Compze.Tessaging/dev_docs/DONE/durable-peer-topology.md`), completing the increment: the one way a peer leaves the endpoint's memory, an administrative act, never an inference (absence — a crash, liveness pruning, a clean stop — never touches peer memory). The act removes the peer from the peer registry — publishes stop fanning out to it, sends stop binding to it, which is also how a retired predecessor's `MultipleHandlersForTessageTypeException` send failure is resolved — and discards everything the endpoint still held for the peer, each delivery tier contributing its share (`IPeerDecommissionParticipant`): the outbox's undelivered tessages awaiting the peer's return and its stranded tommands (their explicit resolution), the best-effort tier's queued tevents and a required peer's first-contact hold. Loud and deliberate: the returned `PeerDecommissionReport` says what was discarded — discarding is part of the explicit act, never a side effect — and the act logs a warning. The whole act is one transaction of its own, with every in-memory consequence deferred to its commit, so an act that fails partway changes nothing. Fails loud on the endpoint itself, on a connected peer (a connected peer is not gone — take it down first), and on an unknown peer. A decommissioned peer that later re-announces is a first contact again: remembered anew from its first advertisement, it receives nothing published before that moment. Registered by the distributed core, so every transport-speaking endpoint has it. Specified in `Peer_decommission_tests` and the two distributed hosting specification classes.
- What the outbox owes a peer follows the peer's own advertisement — the shrink half of increment 6 of the durable peer topology (`src/Compze.Tessaging/dev_docs/DONE/durable-peer-topology.md`). A shrunk advertisement is the peer's explicit renunciation, and the outbox reconciles the peer's undelivered tessages against every replaced advertisement (`Outbox.PeerLifecycleObserver`, notified by the peer registry from inside the transaction that persists the advertisement, always before the peer's connection loads its recovery backlog): undelivered tevents no remaining subscription matches are **discarded, loudly** — they lost their audience by that audience's own choice — and undelivered tommands of types the peer no longer handles are **stranded, loudly**: kept, marked `IsStranded` on the dispatching row (all four SQL backends), and excluded from the recovery backlog, because delivering them would fail on an endpoint that no longer has the handler — someone commanded an action, so they await explicit resolution on the decommission surface rather than deletion. On first contact the outbox discards anything already bound to the peer's identity: nothing can be owed a peer before it is first known, so such rows are leftovers of a decommissioned predecessor (a publish fanning out on the not-yet-replaced peer memory can race a decommission — reconciliation reruns on every replacement, so that leak also self-heals). Specified in `Advertisement_shrink_tests`.
- `DoNotQueueTeventsFor(params EndpointId[])` in the endpoint's composition — the per-peer opt-down from queue-while-down, completing increment 5 of the durable peer topology (`src/Compze.Tessaging/dev_docs/DONE/durable-peer-topology.md`): ephemerality is a property of the relationship, so an endpoint that queues for the peers it depends on can still declare, peer by peer, that it keeps nothing for one it does not care about. A tevent for a not-queued-for peer is delivered only while the peer is connected — published while it is down, it is dropped — and a delivery failure drops the peer's queued stream whole, with the subscriber resuming from tevents published after its return. Mutually exclusive with `RequirePeers`, asserted loud. Specified in `Given_a_met_distributed_tessaging_subscriber_the_publisher_does_not_queue_tevents_for`.
- `RequirePeers(params EndpointId[])` in the endpoint's composition — queue-before-first-contact, continuing increment 5 of the durable peer topology (`src/Compze.Tessaging/dev_docs/DONE/durable-peer-topology.md`): a composition declares the peers it requires by identity, and every tevent published before a required peer's first advertisement is held for it — in order, within the queue bound — with the subset matching its just-learned subscriptions delivered when it is first met. The declaration is the durable peer memory a database-less endpoint cannot keep anywhere else (it survives restarts by being code), and it makes startup deterministic: nothing a required peer should see is lost to the discovery race. The multi-process specification's 30-second publish-retry loop dissolves into one publish — each process requires the other's endpoint (`MultiProcessConversationEndpoints`), so the tevent published before discovery and its reply are both held and delivered on first contact. Specified in `Given_a_distributed_tessaging_endpoint_requiring_a_peer_it_has_never_met`.
- Best-effort tevents queue while a known peer is down — increment 5 of the durable peer topology (`src/Compze.Tessaging/dev_docs/DONE/durable-peer-topology.md`): every remembered subscriber has an in-memory queue on the publisher (`BestEffortTeventQueues`) that outlives connections, fan-out targets the peer registry's remembered subscribers instead of whoever happens to be connected, and the peer's next connection drains its queue in publish order on its return — so a routine restart of a subscriber no longer silently loses every best-effort tevent published in its absence. The drop-stream-whole failure policy is replaced by pause-stream-whole: a delivery failure drops only the one tevent in flight at the failure moment (loudly — without receiver dedup a re-send could duplicate it, and nothing on this tier is ever re-sent) while the remainder stays queued in order, resuming when the peer answers a tessage-free probe (the endpoint-information query) or reconnects. The tier's loss surface shrinks to exactly: that single in-flight tevent, a publisher crash (memory is memory), and queue overflow. The queue bound is 10,000 tevents per peer, and a publish that would exceed it fails loud, naming the peer and the bound (`BestEffortTeventQueueOverflowException`) — backpressure loses nothing while silent shedding does; the bound is enforced by slots reserved at publish inside the caller's transaction, converted on commit and released by a transaction that completes without committing. Specified in `Given_a_met_distributed_tessaging_subscriber_that_goes_down`.
- Every transport-speaking endpoint keeps peer memory — the peer registry moves from the exactly-once vertical into the distributed Tessaging core, with durability following the foundation (increment 5 of the durable peer topology, `src/Compze.Tessaging/dev_docs/DONE/durable-peer-topology.md`): an endpoint whose foundation declares Tessaging persistence remembers its peers in its database (`DurablePeerRegistry` — the previous `PeerRegistry`, renamed for what now distinguishes it), and a database-less endpoint remembers the peers it meets in memory for the life of the process (`ProcessLifetimePeerRegistry`). The registry surface speaks its own type (`RememberedPeer` — identity plus last-known advertisement) instead of the SQL layer's row type, and registration is one plain `IPeerRegistry` per endpoint instead of a component-set that was empty without exactly-once Tessaging.
- "Transient" dies as a name everywhere — increment 4 of the durable peer topology (`src/Compze.Tessaging/dev_docs/DONE/durable-peer-topology.md`). It named a delivery mechanism and leaked into the relationship model, where it is wrong: endpoints conversing without exactly-once guarantees are not transient in their relationship to each other — they depend on each other and typically know each other intimately. The middle composition tier returns to its true name, **distributed**: `AddTransientTessaging` → `AddDistributedTessaging` (`DistributedTessagingComposition`, `DistributedTessagingEndpointFeature`, `DistributedTessagingEndpointComponent`) — in-process ⊂ distributed ⊂ exactly-once, with the distributed tier being where distribution lives (transport, discovery, router) and exactly-once composing it as the guarantee vertical. The delivery rung and leg formerly named "transient" are **best-effort** — the best delivery achievable without transactions, storage, and dedup: `IBestEffortTeventDeliveryLeg`/`BestEffortTeventDeliveryLeg` (namespace `Implementation.BestEffortDelivery`), `BestEffortTeventDirectDispatcher`, `BestEffortTessagingRequestHandlers`, the connection's `BestEffortDeliveryStream`, and — in Compze.Internals.Transport/.AspNet — `TransportRequestKind.BestEffortTevent` and the HTTP route `internal/tessaging/best-effort-tevent`.
- A tommand binds to its one specific receiver at send time, chosen from what the endpoint knows rather than from what happens to be connected — increment 3 of the durable peer topology (`src/Compze.Tessaging/dev_docs/DONE/durable-peer-topology.md`): the live handler when one is connected, otherwise the sole remembered peer whose advertisement handles the type. The payoff (`Tommand_receiver_binding_tests`): a tommand sent while its known handler endpoint is down is delivered when the handler returns — previously the send itself exploded inside the caller's unit of work. Binding keeps every tessage between a sender and a receiver in that pair's single ordered, receiver-deduped delivery stream, so the exactly-once in-order guarantee holds by construction; an in-flight tommand consequently never delivers to a blue/green successor — it waits for its own endpoint — while new sends bind to the live successor. (Routing at delivery time — persist unbound, bind per attempt — was built and retracted the same day: receiver dedup is per endpoint and the received-mark is written after the transport post, so a delivered-but-not-yet-marked tommand could be re-delivered to a successor whose inbox never saw it — handled twice.) No remembered handler fails the send loud (`NoHandlerForTessageTypeException`, whose message now explains what was consulted and that first contact is the boundary); several remembered handlers with none live — a replacement whose retired peer was never decommissioned — fails loud too, naming the peers (`MultipleHandlersForTessageTypeException`). Delivery-starting connections are now looked up at commit rather than at publish/send for tevents and tommands alike, closing a window where a connection appearing in between would have delayed delivery until the next reconnect.
- Exactly-once tevent fan-out reads the peer registry — increment 2 of the durable peer topology (`src/Compze.Tessaging/dev_docs/DONE/durable-peer-topology.md`), closing the rolling-restart data-loss hole: a tevent's receivers are the remembered peers whose last-known advertisement subscribes to it (`IPeerRegistry.SubscriberIdsFor`, matching by the same wrapper-type assignability the router's routes use), never the connections that happen to be live at publish time. A subscribing peer that is down when the tevent is published gets its receiver row persisted anyway, and the recovery backlog its next connection loads delivers the tevent on its return — previously such a tevent was silently lost forever (`Tevent_delivery_to_peers_that_are_down_tests`). Delivery still starts immediately toward the live subscriber connections; enqueue ⊆ persist by construction, since the router records every advertisement before building routes from it.
- The peer registry — increment 1 of the durable peer topology (`src/Compze.Tessaging/dev_docs/DONE/durable-peer-topology.md`): an exactly-once endpoint durably remembers its peers — each peer's identity (`EndpointId`) and last-known advertisement — in its own database (`Peers` + `PeerHandledTessageTypes`, all four SQL backends). The router records every fetched advertisement (first contact creates the peer; a re-fetch replaces its stored advertisement wholesale; the endpoint itself is never a peer), the registry mirrors its storage in memory and loads it in the endpoint's listening phase.
- Tommand and tevent participation handlers receive `IUnitOfWorkResolver` — the context they require, stated in their signature: every path that executes a tommand handler runs it inside a unit of work, and so does every path that delivers a participation tevent handler — the publisher's own unit of work for a local publish (now asserted: `IUnitOfWorkTeventPublisher` throws with no ambient transaction, and `IIndependentTeventPublisher` is the door for such callers), the inbox processing's own for an exactly-once arrival, the direct dispatch's own for a best-effort arrival. Observation handlers keep `IScopeResolver`: their delivery is detached from any transaction by contract. With the publisher asserting its transaction, the best-effort delivery leg always sends on commit — its immediate no-transaction send is deleted as unreachable. `EndpointRequestExecutor` is deleted — its two extension methods were the unnamed independent flavor, hand-rolled; `IIndependentTommandSender`/`IIndependentTeventPublisher` (registered by exactly-once and in-process Tessaging respectively) are the named concept.

- The router's reconciliation is signal-driven: between passes it waits on the registry's change signal (`IEndpointRegistry.AwaitPossibleMembershipChange`), so an endpoint announcing or retracting propagates at signal latency instead of at the old once-per-second poll. The periodic pass remains (once a second) for what no signal can carry: a crashed process's addresses disappearing, and retrying connections that failed.
- Declaring no discovery registry means the endpoint discovers nothing — it serves whatever reaches it, and its own roster serves its sends inline, but it connects to no other endpoint. The configuration fallback (`AppConfigEndpointRegistry`) is deleted: it was a never-implemented placeholder whose `NotSupportedException` crashed any no-registry endpoint at start.
- Tessaging's discovery query lives in Tessaging under its truthful name: `TessagingEndpointInformationQuery`/`TessagingEndpointInformation` (formerly `EndpointInformationQuery`/`EndpointInformation` in Compze.Internals.Transport) — it asks "which remotable tessage types do you handle", is answered only by the Tessaging feature, and consumed only by the Tessaging router, so a style-neutral name in the shared transport assembly was a lying home. Mirrors Typermedia's `TypermediaEndpointInformationQuery` exactly: each style defines its discovery payload in its own assembly, over the shared transport's style-neutral discovery mechanism. The distributed feature maps the type; the shared transport assembly no longer declares any type mappings.
- `NetworkTopologyQuery`/`NetworkTopology` are deleted: an endpoint could answer "whose addresses do you know" from its registry, but nothing anywhere sent the query. If seed-endpoint topology bootstrap is ever wanted (reach one member, learn the membership), it should be designed as part of the cross-machine discovery story, not kept as an unadvertised orphan. Its handler registration was also the only reason the serving side touched the endpoint registry.
- The full pipeline is named for what distinguishes it: the composition originally named `AddDistributedTessaging` became `AddExactlyOnceTessaging` (`ExactlyOnceTessagingEndpointFeature`, `ExactlyOnceTessagingComposition`, and the testing host's `ExactlyOnceTessagingTestingEndpointHostFeature` renamed along with it). With the middle composition tier being genuinely distributed in the plain-English sense, "distributed" no longer distinguished the full pipeline — its distinguishing content is the exactly-once vertical, and the delivery model's own vocabulary (participation / exactly-once / best-effort / observation) is now the composition surface's vocabulary too. ("Distributed" was later reclaimed as the middle tier's own name — see the vocabulary entry above.)
- Guarantee-free Tessaging is a composition of its own: `AddDistributedTessaging(tessaging => tessaging.NewtonsoftSerializer())` (`DistributedTessagingEndpointFeature`) is the transport-speaking Tessaging core — everything in-process Tessaging has, plus the endpoint's one transport server, the router that connects to the other endpoints, and the best-effort tevent delivery leg. It persists nothing, so it composes on the database-less `EndpointFoundation` — no outbox, no inbox, no SQL anywhere: a published `IRemotableTevent` crosses the wire best-effort, and participation and the best-effort tier are all the delivery there is (see `dev_docs/tevent-delivery-model.md`). Tessaging now splits three ways, each layer containing the one below: in-process ⊂ distributed ⊂ exactly-once — `AddExactlyOnceTessaging` composes the distributed core and adds the exactly-once vertical (inbox, outbox, and the tommand senders). Discovery (`DiscoverEndpointsThrough`/`AnnounceAddressTo`/`ParticipateIn`) and the endpoint's `TessagingAddress` belong to the distributed core; the exactly-once feature delegates. Proven across real OS processes with no database in either process (`Given_a_separate_process_hosting_a_distributed_tessaging_endpoint_discovered_through_a_shared_interprocess_registry`).
- A connection carries one ordered send stream per delivery tier the endpoint wires — `TessagingConnection` split into its `BestEffortDeliveryStream` (in-memory, drop-stream-whole; intrinsic to every connection) and `ExactlyOnceDeliveryStream` (durable, head-of-line, storage-backed recovery; carried exactly when the outbox's wiring grants the router its stream factory). The router's delivery lifecycle moved from `Outbox.StartAsync` to the distributed core's endpoint component, where it belongs — the outbox's lifecycle is its storage, now initialized in the listening phase so every connection's exactly-once stream finds it ready when host-wide sending starts.
- The setup-time wiring rule is real: on an endpoint whose composition wires no exactly-once machinery, registering a handler for a tessage type that declares the exactly-once contract fails at setup naming the types — observation subscriptions included, since an observation subscription joins the advertisement and observing a remote exactly-once tevent still requires receiving it exactly-once. An arriving exactly-once tessage on such an endpoint is refused loud (its transport server serves no handler for the request kind), so the tessage stays undelivered on the sender — never silently downgraded.
- Every advertised non-infrastructure type must get a route, asserted loudly on both ends: `HandledRemoteTessageTypeIds` asserts the advertised set's soundness at the advertising endpoint's setup — which also moves the advertisement's TypeId-mapping validation from first-peer-query time to setup — and the router asserts route-by-route while registering. The tommand side is settled by the same assert: Tessaging routes tommands exactly-once only (the best-effort tier is a tevent concept), so a remotable non-exactly-once tommand handler fails loud instead of advertising a silently dead type. Tevent advertisement now filters on the exact shape routing serves (the translated wrapper-of-remotable type), so a wrapper-typed subscription whose inner tevent type is not remotable is truthfully absent from the advertisement instead of advertised dead.
- Tessaging's request handling split per tier: the distributed core contributes the best-effort-tevent handler (`BestEffortTessagingRequestHandlers`), exactly-once Tessaging contributes the exactly-once handlers that receive into the inbox (`ExactlyOnceTessagingRequestHandlers`) — each capability serves exactly the request kinds its wiring can honor.
- Observation — the transaction-ignoring subscription escape hatch — is built (see `dev_docs/tevent-delivery-model.md`): `RegisterTransactionIgnoringTeventHandlers` (on the endpoint builder and the in-process Tessaging feature) registers handlers that fire once, immediately, when a matching tevent is first registered — at publish time for a local publish, at inbox registration (after dedup, before transactional processing) for an exactly-once arrival, on arrival for a best-effort tevent — always in a fresh scope with any ambient transaction suppressed, undeterred by the fate of the transaction the tevent was published or is processed in. A throwing observer is reported through the background-exception reporter, never retried, and never stops the remaining observers or the triggering publish/arrival. An observation subscription joins the endpoint's remote advertisement like any other subscription, so an observer-only endpoint still pulls the tevent across the wire. There is deliberately no publish-side counterpart: code that must emit regardless of its transaction's fate suppresses the ambient transaction and publishes through `IIndependentTeventPublisher`, stating the escape explicitly at the call site (see "No publish-side escape hatch" in `dev_docs/tevent-delivery-model.md`).
- The background-exception reporter is registered by the in-process Tessaging core (observation needs it in every composition) instead of by exactly-once Tessaging, which composes the core.
- The best-effort delivery tier is built end to end (see `dev_docs/tevent-delivery-model.md`): a published `IRemotableTevent` whose type declares no exactly-once guarantee now crosses the wire best-effort. `IUnitOfWorkTeventPublisher` routes it through the endpoint's best-effort delivery leg — on commit, so a rolled-back transaction never leaks a tevent — each subscriber connection carries an in-memory best-effort stream delivering in order while deliveries succeed, and the receiving endpoint dispatches the arriving tevent directly to its handlers in their own scope and transaction, with no inbox, no dedup and no retry. A delivery failure drops the connection's remaining queued best-effort stream *whole* — the unit of loss is the stream, never a single tessage mid-stream — and tessages published afterwards form a new live stream; a failed handling on the receiving side is reported through the background-exception reporter and the tevent is gone. The transport acknowledgement is written after the handlers have executed, so single-in-flight-per-destination keeps handling in send order.
- The router's exactly-once-only tevent routing gate is gone: a route is registered for every advertised remotable tevent subscription, matching stays pure wrapper-type assignability, and which delivery leg a matched tevent travels is decided by the published tevent's own type — not by routing.
- The loud unwired-leg publish failure is real: an endpoint that wires any remote tevent delivery but not the leg a published tevent's contract demands fails the publish with an error naming the missing leg. Zero wired legs remains the deliberately in-process composition, where participation already serves every subscriber that can exist.
- The endpoint-wide tevent publication mode is gone; `IUnitOfWorkTeventPublisher` (in Compze.Abstractions) is the one way to publish a tevent, routing each by the delivery contract its type declares. `InProcessTessagingEndpointFeature` — the style's synchronous core, into which `TessageHandlingEndpointFeature`/`AddTessageHandling` dissolve — registers the publisher; `ExactlyOnceTessagingEndpointFeature` composes it, and wiring the outbox is what wires the exactly-once tevent delivery leg the publisher routes every `IExactlyOnceTevent` through. `InProcessOnlyTeventStoreTeventPublisher`, `DistributedTeventStoreTeventPublisher`, and the in-process/distributed mutual exclusion are removed — declaring both features now composes, exactly as Typermedia's split always has. `IInProcessTeventPublisher` is internal: the in-process delivery behind the publisher, not a public seam.
- Exactly-once Tessaging fails loud at setup when the endpoint's foundation is missing: adding the feature asserts that the transport protocol, Tessaging persistence, and Tessaging serializer are declared, each failure naming the missing declaration — instead of a dependency-resolution error deep inside the container. `ExactlyOnceTessagingComposition` is the feature's compose surface (its serializer slot).
- Tessaging no longer runs a transport server of its own: it contributes its request handling to the endpoint's one transport server (`EndpointTransportServerFeature`, which `ExactlyOnceTessagingEndpointFeature` composes). `IInbox` slims to receiving (`StartAsync`/`ReceiveAsync`) — the inbox no longer owns a server or an address, which also dissolves the inbox/server constructor cycle. `AnnounceAddressTo` now delegates to the shared feature: the announced address is the endpoint's one transport-server address, announced once the whole host is listening.
- Fixed: in-order delivery was lost across a sender restart. Recovery reloaded the undelivered backlog ordered by retry metadata (`RetryCount`, `LastAttemptTime`) — inbox-style thinking misapplied to the outbox — so a tessage stuck retrying at the head of the queue came back last, inverting send order. `GetUndeliveredTessagesForEndpoint` now orders by the outbox tessage table's monotonic `GeneratedId` in every SQL backend, re-establishing head-of-line on the oldest undelivered tessage.
- Named-pipe Tessaging: the named-pipe endpoint transport (`NamedPipeEndpointTransport()`, in `Compze.Internals.Transport`) carries Tessaging cross-process with no ASP.NET Core anywhere in the process. The endpoint's transport server also answers endpoint-discovery queries, so discovery bootstraps through the same address exactly as with HTTP.
- Tessaging's transport is protocol-free: one `TessagingRequestHandlers` contribution receives arriving tessages and one `TransportMessagePoster` posts them, both over the endpoint transport — the named-pipe and HTTP variants of both are gone, and with them the `Compze.Tessaging.Hosting.AspNetCore` assembly. `ExactlyOnceTessagingEndpointFeature` registers both itself; the composing layer declares only the endpoint's transport protocol (`NamedPipeEndpointTransport()` / `AspNetCoreEndpointTransport()`).
- `IInbox` is registered `WithServiceResolver()`: the named-pipe inbox transport server delivers received tessages to the inbox while the inbox owns and starts the server — the constructor cycle is broken by depending on a deferred `IServiceResolver<IInbox>`.
- The in-process bus routes exclusively by wrapper type: `ForTevent<TTevent>` keys an inner tevent type subscription under `IPublisherIdentifyingTevent<TTevent>` and unwraps at delivery, while a subscription to a wrapper type receives the wrapper itself - publisher-conscious subscription (subscribe to `IManagerTevent<IEmployeeTevent>` and receive only the employee tevents a manager published). Subscribing to a tevent type and subscribing to its wrapper receive exactly the same tevents.
- `IUnitOfWorkTeventPublisher.Publish` wraps a tevent published without a publisher-identifying wrapper before routing, and the inbox wraps tevents received from the wire the same way.
- `ForTevent` now validates the subscribed type with the subscription rules (interfaces only) instead of the general message-type rules, matching the in-memory dispatcher.
- Fixed: registering a second tevent handler whose subscription resolves to an already-registered routing key threw instead of appending the handler.
- The wire carries the fully wrapped tevent: the outbox stores and transmits the wrapper under the closed wrapper type's `TypeId`, endpoints advertise tevent subscriptions in their translated wrapper form, and the router matches wrapped tevents against those advertised wrapper types. Publisher identity crosses endpoints with zero information loss - a remote endpoint can subscribe publisher-consciously (to `IMyTaggregateTevent<IMyTaggregateTevent>`) and receive the wrapped tevent exactly as the taggregate published it. Exactly-once deduplication is unchanged: the dedup identity is the wrapped tevent's own `Id`, carried as transport-envelope data.
- `IOutbox.PublishTransactionally` and `ITessagingRouter.SubscriberConnectionsFor` take the wrapped tevent (`IPublisherIdentifyingTevent<IExactlyOnceTevent>` - covariance makes the wrapped tevent statically exactly-once), so handing them an unwrapped tevent - which no wrapper-typed route would ever match - is a compile error instead of a silent routing no-op.
- The publisher-identifying wrapper carries only publisher identity, not delivery-guarantee markers. `IPublisherIdentifyingTevent<out TTevent>` is the sole wrapper interface; the guarantee lives on the wrapped tevent and is read via covariance. The dedup identity for the exactly-once path is the wrapped tevent's own `Id`, extracted once at the outbox entry and carried as transport-envelope data (so the same delivery/storage path serves both tevent-wrappers and tommands). The earlier `IRemotablePublisherIdentifyingTevent<>`/`IExactlyOncePublisherIdentifyingTevent<>` tiers, which mirrored the inner's guarantee interfaces onto the wrapper, are removed.
- Fixed: an endpoint whose multiple subscriptions matched the same tevent was returned once per matching route by `SubscriberConnectionsFor`, so the outbox saved duplicate dispatching rows for it and delivery failed with a primary-key violation. An endpoint is one subscriber however many of its advertised subscriptions match.
