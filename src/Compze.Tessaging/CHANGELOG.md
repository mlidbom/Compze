# Changelog

All notable changes to Compze.Tessaging will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/).

## Unreleased

- The transient delivery tier is built end to end (see `_docs/tevent-delivery-model.md`): a published `IRemotableTevent` whose type declares no exactly-once guarantee now crosses the wire best-effort. `ITeventPublisher` routes it through the endpoint's transient delivery leg ŌĆö on commit when a transaction is present (a rolled-back transaction never leaks a tevent), immediately otherwise ŌĆö each subscriber connection carries an in-memory transient stream delivering in order while deliveries succeed, and the receiving endpoint dispatches the arriving tevent directly to its handlers in their own scope and transaction, with no inbox, no dedup and no retry. A delivery failure drops the connection's remaining queued transient stream *whole* ŌĆö the unit of loss is the stream, never a single tessage mid-stream ŌĆö and tessages published afterwards form a new live stream; a failed handling on the receiving side is reported through the background-exception reporter and the tevent is gone. The transport acknowledgement is written after the handlers have executed, so single-in-flight-per-destination keeps handling in send order.
- The router's exactly-once-only tevent routing gate is gone: a route is registered for every advertised remotable tevent subscription, matching stays pure wrapper-type assignability, and which delivery leg a matched tevent travels is decided by the published tevent's own type ŌĆö not by routing.
- The loud unwired-leg publish failure is real: an endpoint that wires any remote tevent delivery but not the leg a published tevent's contract demands fails the publish with an error naming the missing leg. Zero wired legs remains the deliberately in-process composition, where participation already serves every subscriber that can exist.
- The endpoint-wide tevent publication mode is gone; `ITeventPublisher` (in Compze.Abstractions) is the one way to publish a tevent, routing each by the delivery contract its type declares. `InProcessTessagingEndpointFeature` ŌĆö the style's synchronous core, into which `TessageHandlingEndpointFeature`/`AddTessageHandling` dissolve ŌĆö registers the publisher; `DistributedTessagingEndpointFeature` composes it, and wiring the outbox is what wires the exactly-once tevent delivery leg the publisher routes every `IExactlyOnceTevent` through. `InProcessOnlyTeventStoreTeventPublisher`, `DistributedTeventStoreTeventPublisher`, and the in-process/distributed mutual exclusion are removed ŌĆö declaring both features now composes, exactly as Typermedia's split always has. `IInProcessTeventPublisher` is internal: the in-process delivery behind the publisher, not a public seam.
- Distributed Tessaging fails loud at setup when the endpoint's foundation is missing: adding the feature asserts that the transport protocol, Tessaging persistence, and Tessaging serializer are declared, each failure naming the missing declaration ŌĆö instead of a dependency-resolution error deep inside the container. `DistributedTessagingComposition` is the feature's compose surface (its serializer slot).
- Tessaging no longer runs a transport server of its own: it contributes its request handling to the endpoint's one transport server (`EndpointTransportServerFeature`, which `DistributedTessagingEndpointFeature` composes). `IInbox` slims to receiving (`StartAsync`/`ReceiveAsync`) ŌĆö the inbox no longer owns a server or an address, which also dissolves the inbox/server constructor cycle. `AnnounceAddressTo` now delegates to the shared feature: the announced address is the endpoint's one transport-server address, announced once the whole host is listening.
- Fixed: in-order delivery was lost across a sender restart. Recovery reloaded the undelivered backlog ordered by retry metadata (`RetryCount`, `LastAttemptTime`) ŌĆö inbox-style thinking misapplied to the outbox ŌĆö so a tessage stuck retrying at the head of the queue came back last, inverting send order. `GetUndeliveredTessagesForEndpoint` now orders by the outbox tessage table's monotonic `GeneratedId` in every SQL backend, re-establishing head-of-line on the oldest undelivered tessage.
- Named-pipe Tessaging: the named-pipe endpoint transport (`NamedPipeEndpointTransport()`, in `Compze.Internals.Transport`) carries Tessaging cross-process with no ASP.NET Core anywhere in the process. The endpoint's transport server also answers endpoint-discovery queries, so discovery bootstraps through the same address exactly as with HTTP.
- Tessaging's transport is protocol-free: one `TessagingRequestHandlers` contribution receives arriving tessages and one `TransportMessagePoster` posts them, both over the endpoint transport ŌĆö the named-pipe and HTTP variants of both are gone, and with them the `Compze.Tessaging.Hosting.AspNetCore` assembly. `DistributedTessagingEndpointFeature` registers both itself; the composing layer declares only the endpoint's transport protocol (`NamedPipeEndpointTransport()` / `AspNetCoreEndpointTransport()`).
- `IInbox` is registered `WithServiceResolver()`: the named-pipe inbox transport server delivers received tessages to the inbox while the inbox owns and starts the server ŌĆö the constructor cycle is broken by depending on a deferred `IServiceResolver<IInbox>`.
- The in-process bus routes exclusively by wrapper type: `ForTevent<TTevent>` keys an inner tevent type subscription under `IPublisherIdentifyingTevent<TTevent>` and unwraps at delivery, while a subscription to a wrapper type receives the wrapper itself - publisher-conscious subscription (subscribe to `IManagerTevent<IEmployeeTevent>` and receive only the employee tevents a manager published). Subscribing to a tevent type and subscribing to its wrapper receive exactly the same tevents.
- `ITeventPublisher.Publish` wraps a tevent published without a publisher-identifying wrapper before routing, and the inbox wraps tevents received from the wire the same way.
- `ForTevent` now validates the subscribed type with the subscription rules (interfaces only) instead of the general message-type rules, matching the in-memory dispatcher.
- Fixed: registering a second tevent handler whose subscription resolves to an already-registered routing key threw instead of appending the handler.
- The wire carries the fully wrapped tevent: the outbox stores and transmits the wrapper under the closed wrapper type's `TypeId`, endpoints advertise tevent subscriptions in their translated wrapper form, and the router matches wrapped tevents against those advertised wrapper types. Publisher identity crosses endpoints with zero information loss - a remote endpoint can subscribe publisher-consciously (to `IMyTaggregateTevent<IMyTaggregateTevent>`) and receive the wrapped tevent exactly as the taggregate published it. Exactly-once deduplication is unchanged: the dedup identity is the wrapped tevent's own `Id`, carried as transport-envelope data.
- `IOutbox.PublishTransactionally` and `ITessagingRouter.SubscriberConnectionsFor` take the wrapped tevent (`IPublisherIdentifyingTevent<IExactlyOnceTevent>` - covariance makes the wrapped tevent statically exactly-once), so handing them an unwrapped tevent - which no wrapper-typed route would ever match - is a compile error instead of a silent routing no-op.
- The publisher-identifying wrapper carries only publisher identity, not delivery-guarantee markers. `IPublisherIdentifyingTevent<out TTevent>` is the sole wrapper interface; the guarantee lives on the wrapped tevent and is read via covariance. The dedup identity for the exactly-once path is the wrapped tevent's own `Id`, extracted once at the outbox entry and carried as transport-envelope data (so the same delivery/storage path serves both tevent-wrappers and tommands). The earlier `IRemotablePublisherIdentifyingTevent<>`/`IExactlyOncePublisherIdentifyingTevent<>` tiers, which mirrored the inner's guarantee interfaces onto the wrapper, are removed.
- Fixed: an endpoint whose multiple subscriptions matched the same tevent was returned once per matching route by `SubscriberConnectionsFor`, so the outbox saved duplicate dispatching rows for it and delivery failed with a primary-key violation. An endpoint is one subscriber however many of its advertised subscriptions match.
